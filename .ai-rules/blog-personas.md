## 2. 글 → 페르소나 매핑 선택 기준

신규 글 또는 author 재할당 시 아래 우선순위로 페르소나를 선택한다.

### 2.1 1차: 주제 키워드 매칭

글의 **핵심 키워드 1개**를 정해 카탈로그의 "담당 주제 키워드" 열과 매칭. 정확히 일치하면 그 페르소나.

### 2.2 2차: 핵심 도메인 우선

여러 페르소나가 매칭될 때, **글의 본질(핵심 도메인)** 이 어느 쪽인지로 판단.

- **플랫폼이 본질** → 플랫폼 전문가
  - 예: "ChatGPT로 인스타 캡션 작성" → ChatGPT(도구)지만 본질은 인스타 운영 → `jamie`인가 `ava`인가? **도구 활용법이 본질이면 `ava`, 인스타 캡션 작성 노하우가 본질이면 `jamie`**
  - 예: "캡컷으로 틱톡 영상 제작" → 도구지만 본질은 틱톡 → `noah`
- **업종이 본질** → 업종 전문가
  - 예: "소상공인 SNS 이벤트 전략" → SNS지만 본질은 소상공인 마케팅 → `oliver`
- **트렌드·정책이 본질** → `sophia`
  - 예: "메타 AI 도입" → 인스타지만 본질은 플랫폼 정책·트렌드 → `sophia`

### 2.3 3차: 분포 균형 (참고용)

특정 페르소나에 글이 과도하게 몰리는 것을 피한다. 단, **억지로 분산하지 말 것** — 매핑 정확도가 우선. 분포는 결과의 자연스러운 산물이어야 한다.

### 2.4 매핑 규칙 적용 예시

| 글 제목 | 선택 | 이유 |
|--------|------|------|
| ChatGPT로 인스타 캡션 자동 생성 | ava | 본질은 ChatGPT 활용법 |
| 인스타 릴스 좋아요 3배 늘리기 | jamie | 본질은 인스타 그로스 |
| 캡컷으로 틱톡 숏폼 제작 | noah | 본질은 틱톡 영상 제작 |
| 2025 정부 창업지원금 총정리 | oliver | 본질은 소상공인 지원 |
| GA4로 SNS 유입 트래픽 추적 | mason | 본질은 분석·전환 측정 |
| Meta AI 대전환 인스타 변화 | sophia | 본질은 플랫폼 트렌드 뉴스 |
| 동네 가게 로컬 SEO | lucas | 본질은 로컬 비즈니스 |
| 인플루언서 협업으로 매출 5배 | emma | 본질은 브랜드 협업·UGC |

### 2.5 자동 선택 알고리즘 (Claude 실행용)

`/blog create` 워크플로우에서 페르소나가 명시되지 않은 경우, 메인 키워드·제목을 입력으로 받아 아래 **결정 트리**를 순서대로 적용한다. 첫 매칭에서 멈춘다.

**Step 1 — 트렌드·정책·플랫폼 사태 신호 우선** (다른 도메인보다 우선)
- 키워드에 `메타`, `Meta`, `정책`, `대규모`, `사태`, `정지`, `차단`, `규제`, `업데이트 발표`, `AI 도입`, `알고리즘 변경 발표` 중 1개 이상 + 시점 표현(`2025`, `2026`, `최근`, `이번주`) → **`sophia` (16)**

**Step 2 — 플랫폼 단일 키워드 직매칭**
- `인스타`, `Instagram`, `릴스`, `스토리`, `DM`, `해시태그` (단, 숏폼 제작 본질이면 Step 3로) → **`jamie` (8)**
- `유튜브`, `YouTube`, `쇼츠`(YouTube Shorts 맥락), `썸네일`, `구독자`, `시청 완료율`, `CTR` → **`ryan` (9)**
- `틱톡`, `TikTok`, `숏폼` (제작·기획 본질), `크로스 플랫폼` → **`noah` (12)**
- `트위터`, `X 팔로워`, `X 알고리즘`, `Threads` (단독은 jamie로 흡수) → **`eden` (17)**

**Step 3 — 도메인 본질 판정** (도구·플랫폼 양쪽 키워드 충돌 시)
- `소상공인`, `자영업`, `4대보험`, `세무`, `노동법`, `정부지원`, `창업`, `사업계획서`, `인허가` → **`oliver` (10)**
- `ChatGPT`, `Canva`, `캡컷`, `Zapier`, `노코드`, `자동화`, `AI 도구`, `프롬프트` (단, 플랫폼 운영 본질이면 Step 2 우선) → **`ava` (11)**
- `광고 운영`, `AB 테스트`, `전환`, `GA4`, `랜딩`, `퍼포먼스`, `ROI`, `ROAS`, `CAC` → **`mason` (13)**
- `브랜드`, `UGC`, `인플루언서 협업`, `리뷰 관리`, `팬덤`, `멤버십`, `스토리텔링` → **`emma` (14)**
- `로컬`, `상권`, `동네`, `오프라인 매장`, `네이버 플레이스`, `로컬 SEO` → **`lucas` (15)**

**Step 4 — 충돌·복합 키워드 우선순위**
- "도구 + 플랫폼" 복합 (예: "ChatGPT로 인스타 캡션") → **도구 활용 노하우가 본질이면 `ava`, 플랫폼 운영 노하우가 본질이면 해당 플랫폼 페르소나** (§2.4 참조)
- "거래 의도 + 플랫폼" (예: "인스타 팔로워 구매") → **플랫폼 페르소나 우선** (`jamie`)

**Step 5 — Fallback**
- Step 1-4에서 매칭 0개 → 메인 키워드를 §1 카탈로그 "담당 주제 키워드" 컬럼과 substring 매칭 → 일치 시 해당 페르소나
- 여전히 0개 → `sophia` (트렌드 리포터, 범용 커버)
- 매칭 2개 이상 (Step 2-3 중복 발화) → **§2.2 핵심 도메인 우선 룰**로 본질 1개 선정. 자동 판단 불가능하면 사용자에게 "후보 [A, B] 중 어느 쪽?" 1회 물어본다 (블로커 아님)

**검증**: Phase 1-E에서 알고리즘 적용 결과(예: `jamie (8)`)를 1줄로 산출물에 명시. 사용자가 다른 페르소나를 원하면 `--author={slug}` CLI 옵션 또는 draft frontmatter `author` 직접 수정.

## 3. 카테고리/주제 확장 시 룰

### 3.1 기존 페르소나로 흡수 가능한가 먼저 판단

신규 주제가 들어오면 **카탈로그의 10개 분야로 흡수 가능한지** 먼저 확인. 흡수 가능하면 신규 페르소나 만들지 않는다.

예: "스레드(Threads) 운영법" → 인스타 자매 플랫폼 → `jamie` 흡수 가능

### 3.2 신규 페르소나 생성 기준 (모두 충족 시에만 생성)

아래 **3가지 모두 충족** 시에만 신규 페르소나 생성:

1. **누적 글 수 ≥ 5건 예상** — 신규 분야가 향후 6개월 내 최소 5건 이상 글이 나올 주제
2. **기존 페르소나와 명확히 구분되는 도메인** — 기존 10명 중 어느 누구의 전문 분야로도 자연스럽게 묶이지 않음
3. **읽는 사람의 멘탈 모델에서 별도 카테고리** — 사용자가 author 페이지를 봤을 때 기존 10명과 헷갈리지 않을 만큼 독립적

위 기준에 미달하면 **기존 페르소나의 담당 주제 키워드를 확장**해 흡수.

### 3.3 신규 페르소나 명명 규칙

- **이름 스타일**: 영어 이름을 한글 표기 (예: 제이미, 엠마, 라이언). 한국 본명 사용 금지.
- **slug**: 영어 이름의 lowercase (예: `jamie`, `ryan`)
- **username**: `persona_{slug}` (WP 로그인 ID)
- **email**: `{slug}@personas.helpsns.com`
- **role**: `author` (administrator/editor 금지)
- **description**: 1~2문장. 전문 분야 + 다루는 주제 키워드. 100자 내외.

### 3.4 신규 페르소나 추가 절차

1. 본 문서 §1 카탈로그 표에 행 추가
2. §1.1 담당 주제 키워드 채우기
3. WP REST API로 user 생성 (admin 권한 필요):
   ```bash
   POST /wp-json/wp/v2/users
   {
     "username": "persona_{slug}",
     "email": "{slug}@personas.helpsns.com",
     "password": "<28자 랜덤>",
     "name": "{한글 이름}",
     "slug": "{slug}",
     "description": "{전문 분야 1~2문장}",
     "roles": ["author"]
   }
   ```
4. 응답의 `id`를 §1 표에 기록
5. avatar는 기본값(Gravatar 회색 기본 이미지) 유지. 필요 시 추후 별도 작업.

### 3.5 페르소나 폐기 절차

폐기 사유 발생 시 (예: 분야 통폐합):

1. 해당 페르소나의 글을 다른 페르소나로 재할당 (`POST /posts/{id} { author: 새id }`)
2. WP REST API로 user 삭제:
   ```bash
   DELETE /wp-json/wp/v2/users/{id}?force=true&reassign={fallback_id}
   ```
3. §1 카탈로그 표에서 행 제거
4. 백업 JSON은 `.recycle/`에 보관

## 3.5 작성자 페이지 E-E-A-T 2026 필수 (CRITICAL)

seo-policy.md §1.7 E-E-A-T 2026 March update에 따라 페르소나 author 페이지가 반드시 다음 요소를 포함해야 한다 (미달 시 audit −5):

**WP user description (또는 `src/shared/data/personas.ts`) 필수 필드**:
- **전체 이름** (예: "에이바 김", "엠마 박") — 한국 시장 친화 한글 표기
- **전문 헤드샷** (스톡 이미지 X, 일관된 페르소나 일러스트 OK — `wp-content/illustrations/persona-{slug}.webp`). 본 페르소나 일러스트는 author 페이지·byline E-E-A-T 신호 자산이며, asset-images §4.8.6 "본문 단독 인물·장면 일러스트 금지" 룰 적용 범위에서 **제외**된다. byline·author page 노출용 자산은 `persona-{slug}.webp`로 라이브러리에 유지하고, 본문 `<figure>` 단독 일러스트와 명확히 구분한다.
- **자격증·학위·경력 연차** ("AI 마케팅 자동화 5년 운영", "CMP 자격증 보유")
- **LinkedIn 또는 검증 가능한 외부 링크** (페르소나 가상이라면 brand 도메인 author page URL `/author/{slug}/`)
- **도메인 전문성 명시** (`knowsAbout` schema 배열에 키워드 5-10개)
- **글 인덱스** (그 페르소나가 쓴 모든 글 페이지)

**Person schema 출력 (자동, src/pages/author/[slug]/index.astro)**:
- `jobTitle` (전문 분야)
- `knowsAbout` (담당 주제 키워드 배열)
- `sameAs` (LinkedIn·X·검증 외부 URL)
- `worksFor` (Organization #organization 참조)

**페르소나 정보 노출 위치 (seo-policy §1.9 글 상단 구조 정본 연결, CRITICAL)**:

페르소나 시그널은 **본문 markdown에 자기소개 단락으로 박지 않는다**. E-E-A-T 시그널은 다음 3중 노출로 충분하다.

| 노출 위치 | 처리 주체 | 비고 |
|---|---|---|
| (a) 페이지 상단 byline | Astro 레이아웃 자동 ([blog/[slug]/index.astro:541-547](../src/pages/blog/[slug]/index.astro#L541-L547)) | `rel="author"` author page 링크 포함 |
| (b) 글 말미 "저자 소개" 박스 | Astro 레이아웃 자동 (`src/widgets/blog/AuthorBox.tsx` — frontmatter author + personas.ts 정보 기반) | 사진·자격·LinkedIn·전문 분야 노출 |
| (c) Person JSON-LD schema | Astro 레이아웃 자동 (line 428-444) | sameAs·image·credential |

**금지 (seo-policy §1.9 정본 룰)**:
- ❌ 본문 첫 단락에 "이 글은 SNS헬프 X가 N년..." / "안녕하세요 저는..." / "X는 ... 전문가로 N년..." 자기소개 작성
- 사유: GEO/AI 인용 가치 손실 + 첫 100자 keyword density 낭비 + 검색 사용자 이탈률 ↑. E-E-A-T 시그널은 (a)+(b)+(c)로 동일하게 전달됨

**Experience 신호는 본문 안에서 다음 형태로 노출** (자기소개 단락이 아닌 본문 자체에 녹임):
- 섹션 안 first-hand 디테일: "Sora 1.5로 5초 클립 12편 만들어보니..." (작성자 경험 직접 인용)
- 측정값·사례: "2025년 8월 운영 캠페인에서 도달 230,000" (구체 수치)
- 실패·교훈: "처음엔 60초 통째 시도했다가 캐릭터 일관성 깨졌다"
- 좋은 예: 본문 H2 안에 "에이바가 ChatGPT로 캠페인 100개 자동화하며 가장 자주 막힌 지점은 ..." 형태로 자연스럽게 페르소나 경험 인용
- 나쁜 예: 본문 첫 단락 통째로 "에이바는 ... 전문가로 ..." 자기소개

## 4. 보안·운영 주의

- WP 로그인 username (`persona_{slug}`) 및 email은 공개되지 않으나, **WP admin 접근 권한자(administrator/editor)에게는 노출**됨
- `persona_{slug}` 형식이 어드민에 노출되는 것이 부담이면 username을 다른 명명 규칙으로 변경 가능. 단 username은 한 번 정하면 WP에서 변경 불가 (재생성 필요)
- 페르소나의 application password는 발급하지 않는다. WP REST API 자동화는 admin/editor 토큰만 사용
